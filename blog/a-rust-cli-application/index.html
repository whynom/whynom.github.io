The mv commandüîó

The mv command moves a file or directory from one place to another. The rsync command does the same thing, but it can also show progress which is particularly useful for moving things between a local computer and a server. Rsync can also sync things in such a way that if a certain move is taking a very long time it can be halted and picked up later.

I'm going to make a mvto command that's a Rust application that basically does rsync -avz --progress --delete-source-files thing-to-move destination.

Why Rust?üîó

Because it's what I'm trying to learn, and there's no better way than doing. This could probably be done with a simple shell script, or maybe an alias, but as I add more and more to it, there's a possibility that the crustiness of bash will start to outweigh the added complexity of doing something dead simple in a full programming language like Rust.

A command line programüîó

use std::process::Command;

fn main() {
    let output = Command::new("ls")
        .output()
        .expect("Failed to execute command");

    println!("stdout: {}", String::from_utf8_lossy(&output.stdout));
}
  1. The Command struct

That import at the top gives us access to the Rust struct Command. According to the documentation.

A process builder, providing fine-grained control over how a new process should be spawned.

The term process as it's used here means process your computer runs. In my parlance, I like to think of it as it'll do a command line command. It'll run ls or rm as a child command and can take the output from those commands and do stuff with it in Rust.

  1. Command::new("ls").output()

This just runs the ls list command sets the output of that command to the output variable.

  1. println!("stdout: {}", String::from_utf8_lossy(&output.stdout));

This prints the output from the ls command we've run as a child process of our Rust application

Adding some argumentsüîó

The only change here is adding the arg methods to the command to give our ls command some arguments.

fn main() {
    let output = Command::new("ls")
        .arg("-l")
        .arg("-R")
        .output()
        .expect("Failed to execute command");

    println!("stdout: {}", String::from_utf8_lossy(&output.stdout));
}

But when making our eventual rsync like command, we'll have to feed in some arguments during the running of the application that can't be hardcoded into the program like -l and -R are here. To do this, we'll have to capture user inputs.

Capturing user inputsüîó

We need to pull in the env module to collect the arguments from the command line and import Path to double check the directory we're trying to list is in fact a directory.

use std::env;
use std::path::Path;

To collect all the arguments given we use the collect method as follows.

    let args: Vec<String> = env::args().collect();
    let first_arg = &args[1];

The collect method actually takes in the application name's on the cli as the first argument, but because I'm talking about the arguments given to the application, I'm calling the first_arg the second string in the collection. I think it's not the right thing to do, but here we are.

We then want to check to make sure we have a directory to feed to our ls command.

    let path = Path::new(&first_arg);

We'll do the ls command on the directory if it is, but if not, we'll just say it's not a directory.

    if path.is_dir() {
        println!("{} is a directory", first_arg);

        let output = Command::new("ls")
            .arg(&first_arg)
        // .arg("-R")
            .output()
            .expect("Failed to execute command");

        println!("stdout: {}", String::from_utf8_lossy(&output.stdout));
    } else {
        println!("{} is not a directory", first_arg);
    }

As the program is now it can take a directory as a user input but if no directory is given, it'll ungracefully error. We could make it ls the local directory when run without an argument, but enough playing around with ls, let's start with the rsyncing.

Very basic rsync functionality.üîó

We're getting rid of the directory checking, so we've removed the Path import and the if statement logic. We change the command to rsync, add the desired arguments and voila, we've got a very basic rsync command.

use std::process::Command;
use std::env;

fn main() {
    let args: Vec<String> = env::args().collect();

    let source = &args[1];
    let destination = &args[2];


    let output = Command::new("rsync")
        .arg("-avz")
        .arg("--progress")
        .arg("--remove-source-files")
        .arg(&source)
        .arg(&destination)
        .output()
        .expect("Failed to execute command");
}

The spawn method and showing the rsync progress.üîó

While the above works, it won't show a progress bar. The reason for making this application is to more or less improve upon the standard mv command, and showing that progress bar was one of the improvements

fn main() {
    let args: Vec<String> = env::args().collect();

    let source = &args[1];
    let destination = &args[2];


    let output = Command::new("rsync")
        .arg("-avz")
        .arg("--progress")
        .arg("--remove-source-files")
        .arg(&source)
        .arg(&destination)
        .spawn()
        .expect("Failed to execute command");
}

This shows the progress, but after it runs, it won't quit the rust application.

Looking at the Command::spawn documntation, we get the following explanation.

Executes the command as a child process, returning a handle to it. By default, stdin, stdout and stderr are inherited from the parent.

What I think this means is that it's inheritance of stdout means it shows this new child process in our terminal and thus shows the progress bar. Success but the application hangs.

When using this application now it completes the moving of the file but the application doesn't stop, it just keeps going.

Progress and quitingüîó

Let's make our command a mutable variable and run the wait method on it.

fn main() {
    let args: Vec<String> = env::args().collect();

    let source = &args[1];
    let destination = &args[2];


    let mut cmd = Command::new("rsync")
        .arg("-avz")
        .arg("--progress")
        .arg("--remove-source-files")
        .arg(&source)
        .arg(&destination)
        .spawn()
        .expect("Failed to execute command");

    let status = cmd.wait().expect("Failed to wait for rsync");

}

After running the rsync command our application ends, as we want.

std::process::Child documentation has the following note.

Waits for the child to exit completely, returning the status that it exited with.

My meager understanding leads me to believe that when we run the spawn method it produces a child process that never stops. The wait method gives us some result from that process thereby ending the the running of our code.

Either way, we've got what we want.

Was it worth it? What's next?üîó

This very easily could have been an alias in a .bashrc but making a Rust program allows us to expand further on our usecase and go beyond what rsync offers. My next goal will be to make our mvto command remove the empty directories that get left behind when using --remove-source-files.

I'm a little reluctant to play around with destroying things, and I suspect this is why rsync doesn't have it built in but it's what I want and Computers should do exactly what I want, not just what other programmers have given me.
