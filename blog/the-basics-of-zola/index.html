Zola🔗

Zola is a static site generator written in the Rust programming language. I've used Hugo a lot before but I always felt like I could barely stumble together a website, and after leaving a Hugo website for a while, I'd come back and it wouldn't work. This is probably more due to something I was doing rather than whatever was going on with Hugo but I thought I'd change it up just because I might as well.

I should also point out, I've never used any of the Go underpinnings of Hugo. I've never used Go, I don't understand it's semantics or anything, so whatever programming language a static site is based upon makes no difference to me. The confusion for me always lay at the templating level of static sites. I decided to go with Zola because all else being equal, I'll take the static site generator written in a language I'm trying to learn. Why not?

Well it turns out there's a huge benefit to using a static site generator written in Rust; the error outputs. Rust has really good and easy to read errors, and using Zola, it also gives really good easy to read error outputs. Awesome.

I've used Hugo in the past, and while I like it, I never quite understood all of the templating and use of blocks and where blocks got replaced and all that. I was able to limp my way through everything with a mix of minimal understanding and trial and error, but I never quite understood everything. After going through Zola's documentary overview with my AI buddy, I finally understand.

Sections🔗

This was initially very confusing to me. What is a section? I think it's best understood with a solid example.

I want my webpage to have several pages. The initial landing page, but I'd also like a blog page. That blog page is going to have the title and the first 5 lines of my 5 most recent blog posts, or maybe my 3 most popular blog posts, or maybe even the whole of my 3 most recent blog posts , but the point is it will contain links to other pages and I have to get my hands on something that can represent the collection of all my blog posts so I can tell Zola which ones I want to show, in what order and to what extent (like the whole post or just a summary with a title).

That collection of all my blog posts is called a section.

A section is really a collection of pages. Although when I go to the my-website.com/blog page, that is a page, when it shows me some collection of my posts, of the type blog, it's showing me some collection of my blog section.

That collection of my blog section can be the 5 most recent posts, my 5 most popular posts, a random smattering of blog posts, or whatever I want, but to be able to organize and filter and sift through all my blog posts and show them on my blog page, I need a some collection of them, and that collection of all my blog posts is a section.

templates/base.html🔗

Our templates directory, unsurprisingly, holds all our templates. This is probably the part that confused me the most, so let's take a look at our base.html.

<!DOCTYPE html>
<html lang="en">

<head>
  <meta charset="utf-8">
  <title>MyBlog</title>
</head>

<body>
  <section class="section">
    <div class="container">
      {% block content %} {% endblock %}
    </div>
  </section>
</body>

</html>

The thing that confused me about this is those block variables. I know they aren't variables, but that's how I was thinking about them. My idea was that base.html as the thing we serve and see, and we put other code into those block. While this is kind of true, really we don't ever actually serve base.html rather it's a recipe for other things we serve, and we put whatever difference we want in that recipe in those code blocks.

Typing this out, it's hard to explain my misunderstanding, but there it is.

templates/index.html🔗

{% extends "base.html" %}

{% block content %}
<h1 class="title">
  This is my blog made with Zola.
</h1>
{% endblock content %}

Maybe it's the extends keyword that confused me. I felt like this templates/index.html just adds more to to the templates/base.html file, but really it's using the base.html file as a template to make it's own index.html template.

This file will become our index page for the whole website or the thing we see first when we visit the website at its domain. This is what we see when we go to mywebsite.com

content/blog/_index.md🔗

The content directory is where we get all our stuff to be served up. Each directory within the content directory represents a collection, or a section, of pages if that directory has a _index.md file.

This file that makes a directory into a section gives us basic information about the section. Our content/blog/_index.md file should look as follows.

+++
title = "List of blog posts"
sort_by = "date"
template = "blog.html"
page_template = "blog-page.html"
+++

The title is giving us the title of the section. The sort_by is giving us the order in which this collection of pages, that is the pages within our content/blog directory, will be given to us when we call upon it.

The template is saying what we will serve when we go to the blog url ourwebsite.com/blog.

The page_template is giving us the template we'll use whenever we serve up a blog like ourwebsite.com/blog/my-post-about-zola.

templates/blog.html🔗

{% extends "base.html" %}

{% block content %}
<h1 class="title">
  {{ section.title }}
</h1>
<ul>
  <!-- If you are using pagination, section.pages will be empty.
       You need to use the paginator object -->  
  {% for page in section.pages %}
  <li><a href="{{ page.permalink | safe }}">{{ page.title }}</a></li>
  {% endfor %}
</ul>
{% endblock content %}

This is the template for the blog page mywebsite.com/blog, the page that's going to show some collection of our blog posts. That collection can be the 5 most popular posts, the first 3 lines and the title of our 3 most recent posts, the whole of our 5 oldest posts, whatever we want.

It's also using the base.html for how it should look because base.html is giving us the very most basic webpage. Just some meta data and the title of the blog and some organization. Instead of having to type out that html in the base.html file over and over again, we've got a little template that we pretty much make every page within our website with. It's convenient that way.

The section.pages is giving us access to our blog section and the pages within that section. How does this page know that the blog section is the one we want to get the pages from?

It knows from the TOML information we have in our content/blog/_index.md file.

templates/blog-page.html🔗

The following is how every one of our blogs in our /content/blog directory will be served up when we visit the page mywebsite.com/blog/a-post-about-zola.

{% extends "base.html" %}

{% block content %}
<h1 class="title">
  {{ page.title }}
</h1>
<p class="subtitle"><strong>{{ page.date }}</strong></p>
{{ page.content | safe }}
{% endblock content %}

An actual blog post🔗

+++
title = "My first post"
date = 2019-11-27
+++

This is my first blog post.

If you look at our templates/blog-page.html which our content/blog/_index.md references as the template for all our blog posts, it refers to our page's titel and date and content. Well that frontmatter at the top of our blog post tells that template the information we need to put page.title and page.date into our post.

The content is everything after the second set of +++.

Updating templates/index.html🔗

We want to actually see our blog posts when we visit the webpage's initial webpage at mywebsite.com so we'll update the templates/index.html file as follows.

{% extends "base.html" %}

{% block content %}
<h1 class="title">
  This is my blog made with Zola.
</h1>
THING I CAN'T PUT
{% endblock content %}

Thing I can't put is a line that, when rendered by this zola website messes up the parsing, so I left it out. There's some way to escape it, but I don't wanna figure that out, and that special block of code can be found at Zola's documentary overview.

Summary🔗

There's a bunch of templates, and section definitions referencing various templates and html being referenced within in templates that aren't server but are they themselves recipes and... just a bunch of confusion.

Writing this has been a very good way for me to get through that confusion so I have a basic understanding of the structure of not just a Zola static web generator but basically most static website generators.

I doubt this could possibly help anyone besides myself, but hopefully, if I ever have any confusion, I can look back on this to try to understand the stuff a little better.
