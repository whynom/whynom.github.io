My quick foray into the Rust programming language.🔗

There is no end of people that will sing the praises of Rust, and not without reason. I do not care too much for the common things many seem to go gaga about, namely memory management that avoids a whole category of errors and small binary sizes. I don't care about these things because I've never run into these problems. This is probably because I've never worked on a big complicated codebase.

There are several things I did go gaga about: extremely clear error messages, an overbearing warning system and a quick and easy way to whip up a new program with testing. That last point is what I'm going to get into with Swift.

XCode you are but my enemy🔗

I like emacs. I hate XCode. XCode forces me to use a bunch of GUI checkboxes to tick in favor of direct text I can more easily examine and debug. I also can't customize it nearly to the degree to which I can customize emacs.

This added up to me wanting to just quickly whirl up a program to test some code outside of whatever application I was working on being a nuisance of making a new XCode project, going through its rigmarole and then finally ending up with having to produce my quick code and tests in an IDE I do not favor.

What I aim to do here is show a way to clearly and quickly setup a space for testing code with swift-testing so I can play around and test out code in the text editor of my choice, emacs.

Whip up that basic application🔗

First let's make a new directory for our app I'm going to call quickApp and go into that new directory and make our new application.

mkdir quickApp
cd quickApp
swift package init --type executable

This gives us a Swift application that we can run with swift run while we are in our quickApp root directory. When we do so we get greeted with the classic "Hello, world!". So far so good.

The given boilerplate code🔗

If you do a quick ls you can see that our quickApp directory has a Sources directory with a main.swift file. The code in that file is where we get our "Hello, world!" print. This file is where we put all the code we want to run.

The other file you'll find in your quickApp directory is Package.swift. This is where we define our application setup like targets and dependencies. This is where I will sing the praises of Rust yet again. The Rust setup file is a very straight forward .toml file. The Swift Package.swift setup file is decidedly more complicated, especially for a quick and dirty setup that we're going for. Let's explore it now.

// swift-tools-version: 6.1
// The swift-tools-version declares the minimum version of Swift required to build this package.

import PackageDescription

let package = Package(
    name: "quickApp",
    targets: [
        // Targets are the basic building blocks of a package, defining a module or a test suite.
        // Targets can depend on other targets in this package and products from dependencies.
        .executableTarget(
            name: "quickApp"),
    ]
)

It's not super complicated but it wasn't trivial for me to setup with just the swift-testing library import and implementation.

Package.swift🔗

Let's add the following to our Package.swift file to target MacOS.

...
name: "quickApp",
platforms: [
  .macOS(.v10_15)
],
...

I'm targeting the mac platform because I'm using a mac and I want to run code quickly and iteratively. I don't want to pop open an iOS simulator or anything like that. We're not looking to make a cross platform amazing program, we're trying to make some kind of very quick application for which we can write tests right off the bat and test things.

Now add the swift-testing dependency.

...
let package = Package(
    name: "quickApp",
    platforms: [
        .macOS(.v10_15)
    ],
    dependencies: [
        .package(url: "https://github.com/apple/swift-testing.git", branch: "main")
    ],
    targets: [
        .executableTarget(
...

We can do a quick swift run and the dependency will be downloaded and put into our program. There will be a warning

warning: 'quickapp': dependency 'swift-testing' is not used by any target

This makes sense because we haven't added the dependency to our testing target yet. Let's do that now.

targets🔗

Before actually adding the Tests target, a quick word about targets. They are basically a way of grouping code. There's the executable target, which is the code for the actual program you're writing, and we're going to add another large group of code separate from the code we're actually writing for the application. We're adding code which will be used for testing and will not be usable in the actual executable code.

This has the benefit of separating our test functions from the functions we actually want to run in the program. I actually personally don't like this because it adds a level of complication that could just be solved by naming testing functions appropriately with test in their name or by making tests have to be run in an instantiation of a testing class or something, but I'm sure there are a million reasons why I'm wrong about that.

So what we want in the end is for the testing code to not be executable in our executable code but we do want our executable code to be runnable in our testing target because we need the tests to be able to run those functions to test them.

Testing target🔗

Let's add a tests directory for our testing target.

mkdir Tests

Now we can add our testing dependency.

...
            name: "quickApp"),
        .testTarget(
            name: "quickAppTests",
            dependencies: [
                "quickApp",
                .product(name: "Testing", package: "swift-testing"),
            ],
            path: "Tests"
        ),
...

A few notable things.

First, you'll notice that our new test target has a dependency and it's the quickApp itself. As stated before our testing code needs to be able to run our executable's code to be able to test that code.

Next, you'll see that we've defined a path for our testing files. It's in the Tests directory we made before. We don't have this path in our executable target because we have all of our code for executable in our quickApp/Sources directory which is the default.

Lastly, we say we want to have access to the swift-testing package so we can use that library in our testing files.

Making and running basic tests🔗

Add a file for our tsting code in our tests directory.

cd Tests
touch quickAppTests.swift

When I'm making a new testing file I like to test the testing file by having a test that should fail, and a test that should pass. Add the following code to the new quickAppTests file.

import Testing

@Test func helloIsNotGoodbye() throws {
    let helloString = "Hello, world!"
    let goodbyeString = "Goodbye, world!"

    #expect(helloString == goodbyeString)
}

@Test func oneIsOne() throws {
    #expect(1 == 1)
}

Change to the quickApp root directory and run our tests

swift test

You should get a result in your terminal telling you that our helloIsNotGoodbye function failed and our oneIsOne function passed.

Conclusion🔗

This is quite a bit of work for doing something that should be a simple command line command but it gets me the desired result, a place I can put code and a place I can test that code. I can also do this all in emacs which I'm constantly in anyway, so I don't have to mess around with XCode.

I also feel like this is still simpler than the XCode route that requires me to check boxes, possibly uncheck certain boxes, and gives me bunch of UI boilerplate code in a few files I'll never need.

In the end, I'd like to make a script that does all of the above, or even possibly a swift application that does because my distate for XCode does not extend to the actual Swift code I feel comfortable with and enjoy writing.
